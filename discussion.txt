## 1. THE VISION: FULLY AGENTIC OS CONTROL

A truly agentic mobile companion must replicate the raw multi-tool orchestration of desktop CLI developers, but inside a gorgeous, unified, and immersive mobile interface. 

### Comparison: Generic Bot vs. VoxKage Agentic Engine
* **Generic Bot:** User asks for 3 things -> Bot addresses the first thing, forgets the rest, or writes a text wall saying what it *will* do without executing.
* **VoxKage Agentic Engine:** User asks: *"Look for the latest FitGirl repack games, gather details, SMS my brother that it is out, check my phone battery, and add a calendar event."* -> The model parses this query into an execution graph, runs web scraping, phone API tool calls, and messaging sub-agents concurrently or in sequence, and updates the user visually at each stage.

## 3. PILLAR A: THE AGENTIC GRAPH ROUTER & PARALLEL EXECUTION (BACKEND)

Currently, `backend/agent_loop.py` handles tool calls in a linear loop: one tool call is returned by the LLM, the python loop runs it, feeds it back to the model, and repeats until the model decides to stop. 

### Proposed Upgrade: Graph Execution Plan
To support parallel and complex multi-tasking, we will restructure the agent loop to utilize structured graph routing:

1. **Pre-Planning Stage:** The model receives the user request and generates an execution plan consisting of multiple tasks:
   * **Task 1:** WebSearch (Query: "Fitgirl repacks latest releases 2026")
   * **Task 2:** System Status (Battery check)
   * **Task 3 (Dependent on 1):** SMS Message (Recipient: Brother, Content: "Game X is out")
   * **Task 4:** Calendar Event (Title: "Interview", Time: "June 21, 5:00 PM")
2. **Concurrency Manager (`asyncio.gather`):** Tasks that are independent (e.g. WebSearch, Battery check, Calendar event) are dispatched **in parallel** to maximize speed.
3. **Dependency Resolver:** Tasks that require outputs from previous tasks (e.g. SMS needs the game name from the FitGirl search) are queued to execute immediately after their dependencies complete.

### Python Backend Code Structure (`backend/agent_loop.py` mock-up)
```python
import asyncio
from typing import List, Dict, Any

class AgentExecutionGraph:
    def __init__(self, plan_nodes: List[Dict[str, Any]]):
        self.nodes = plan_nodes  # List of node dicts with dependencies
        self.results = {}
        self.active_tasks = {}

    async def execute_node(self, node_id: str, ws_connection):
        node = self.nodes[node_id]
        
        # 1. Wait for dependencies to finish
        for dep in node.get("dependencies", []):
            while dep not in self.results:
                await asyncio.sleep(0.1)
        
        # 2. Extract parameters, substituting dependencies if needed
        resolved_args = self.resolve_parameters(node["args"], self.results)
        
        # 3. Notify client: Tool starting
        await ws_connection.send_json({
            "type": "tool_start",
            "node_id": node_id,
            "tool_name": node["tool_name"],
            "label": node["label"]
        })
        
        # 4. Run tool
        try:
            result = await execute_tool(node["tool_name"], resolved_args)
            self.results[node_id] = result
            
            # 5. Notify client: Tool success
            await ws_connection.send_json({
                "type": "tool_success",
                "node_id": node_id,
                "result_summary": summarize_result(result)
            })
        except Exception as e:
            self.results[node_id] = {"error": str(e)}
            # Notify client: Tool failed
            await ws_connection.send_json({
                "type": "tool_failed",
                "node_id": node_id,
                "error": str(e)
            })

    async def run(self, ws_connection):
        # Dispatch independent nodes in parallel
        tasks = [
            asyncio.create_task(self.execute_node(node_id, ws_connection))
            for node_id in self.nodes
        ]
        await asyncio.gather(*tasks)
```

## 4. PILLAR B: THE WEBSOCKET PROGRESS PROTOCOL (COMMUNICATION)

To support real-time UI updates, the WebSocket connection in `backend/main.py` needs to stream granular events.

### Streamed Message Types
* `tool_plan`: Sent at the start of the query to outline the workflow. Contains the list of nodes, labels, and connections.
  ```json
  {
    "type": "tool_plan",
    "nodes": [
      {"id": "node_1", "label": "Search Fitgirl Repacks", "tool": "web_search"},
      {"id": "node_2", "label": "Check Battery Status", "tool": "mobile_get_battery"},
      {"id": "node_3", "label": "SMS Brother", "tool": "mobile_send_sms", "dependencies": ["node_1"]},
      {"id": "node_4", "label": "Schedule Calendar Event", "tool": "mobile_add_calendar"}
    ]
  }
  ```
* `tool_start`: Emitted when a specific node begins execution.
* `tool_success` / `tool_failed`: Emitted when a node completes, enabling the frontend to check off items in the SVG graph.
* `agent_thought`: Streams the model's inner thoughts to be displayed inside the collapsible accordion drawer.

## 5. PILLAR C: SVG WORKFLOW PATHING COMPONENT (FRONTEND)

Instead of outputting text logs, the chat feed will display a custom React Native component: `ToolWorkflowPath`.

### Visual Design
* A clean, vertical or horizontal timeline rendered using React Native SVG (`react-native-svg`).
* Uncompleted tasks are drawn as empty circles with 1px gray borders.
* Active tasks feature an animated spinning glow or a pulsing blue circle.
* Completed tasks render a solid green/blue circle containing a checkmark (`✓`).
* Elegant SVG connector lines connect the circles. As tasks complete, the line colors transition from dark gray to bright blue to visually show the path of execution.

```
       [Start]
          │ (completed path - bright blue)
         (✓) Search FitGirl Games
          │ (completed path - bright blue)
         (✓) Check Battery Status
          │ (active path - pulsing glow)
        (⚙️) SMS Brother
          │ (pending path - dark gray line)
         ( ) Schedule Calendar Event
```

### React Native Component Mockup (`frontend/src/components/chat/ToolWorkflowPath.tsx`)

## 6. PILLAR D: COLLAPSIBLE THINKING ACCORDION (UNIFIED UI)

Instead of cluttering the chat history with separate "thinking logs," we will integrate a unified, collapsible accordion component at the **very top** of the assistant's response bubble.

### Interaction Details
1. **Collapsible Header:** A sleek header bar displaying:
   * A subtle loader spinner.
   * A pulsing label showing the active step: e.g., `Searching for TCS office...` or `Checking battery status...`.
   * A toggle arrow (caret) to expand/collapse.
2. **Deep-Dive Drawer/Popup:** Clicking on the thinking header expands it in place (as an accordion) or opens a gorgeous modal sheet containing the complete, timestamped history of the agent's thoughts:
   * `[13:42:01] Parsing user intent: Multi-task workflow detected.`
   * `[13:42:02] Running 3 parallel searches for TCS offices in Gujarat...`
   * `[13:42:04] 2 matches found. Executing geolocation mapping...`
3. **Auto-Minimize:** Once the agent completes all tasks, the thinking section automatically collapses into a subtle, single-line indicator (`✓ Viewed 3 sources & executed 2 tools`) at the top of the message bubble, leaving a clean, distraction-free response.

## 7. PILLAR E: SHIFTING TO COMPONENT-CENTRIC RESPONSES

To achieve the premium aesthetic requested by Sir, we must restrict plain text responses and enforce the usage of rich, immersive custom components.

### 1. Carousel Cards (e.g. search results, articles, game repacks)
Instead of a bulleted list of links, we will group items into a horizontally scrollable Carousel of cards featuring:
* Domain favicon/source brand indicator.
* Bold title with clean typography.
* Text snippet.
* Premium action buttons (e.g., "Open Link", "Scrape Context", "Share").

### 2. Live Resource Previews
If the agent fetches data from a file or webpage, it will display a file icon component showing file info (similar to the spreadsheet example: `File icon + Title + Type + Size` in a rounded gray pill).

### 3. Customized Map / Location Cards
For location queries (like TCS offices), it should show a simulated mini-map interface with marker pins and an button to launch deep-linked map routes natively, avoiding plain-text lists of addresses.
